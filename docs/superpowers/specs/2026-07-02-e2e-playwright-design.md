# Дизайн: e2e-тесты на Playwright

Дата: 2026-07-02
Статус: согласован, готов к плану реализации

## Краткое описание

В проекте нет e2e-покрытия вообще — только unit-тесты сервера (Node test runner) и
клиента (vitest, только чистая логика). Добавляем `packages/e2e` — новый pnpm-workspace
с Playwright, гоняющий реальный клиент против реального сервера и покрывающий основные
пользовательские сценарии: базовый CRUD списка, мультисписки, шеринг/мультиустройство,
offline-sync.

## Архитектура

### Пакет и запуск

Новый workspace `packages/e2e` (`@kupi/e2e`, private), зависимость — `@playwright/test`.
Playwright поднимает сервер и клиент сам через `webServer` (массив из двух записей в
`playwright.config.ts`), на выделенных портах — **3100** (сервер) и **5174** (клиент), не
пересекающихся с портами `pnpm dev` (3000/5173). Это принципиально: если бы Playwright по
умолчанию переиспользовал уже запущенный `pnpm dev`, тесты работали бы против настоящей
рабочей БД разработчика.

```ts
webServer: [
  {
    command: 'pnpm --filter @kupi/server dev',
    url: 'http://localhost:3100/api/health',
    env: { PORT: '3100', DB_PATH: tmpDbPath },
  },
  {
    command: 'pnpm --filter @kupi/client dev -- --port 5174',
    url: 'http://localhost:5174',
    env: { VITE_API_PROXY_TARGET: 'http://localhost:3100' },
  },
],
use: { baseURL: 'http://localhost:5174' },
projects: [{ name: 'chromium', use: devices['Desktop Chrome'] }],
```

`tmpDbPath` — файл в `os.tmpdir()` (`kupi-e2e-<timestamp>.db`), сгенерированный в
`playwright.config.ts` на каждый прогон. `globalTeardown` удаляет его вместе с
`-wal`/`-shm`/`-journal`-компаньонами после завершения прогона.

### Точечные правки существующего кода

Две инфраструктурные правки нужны, чтобы сервер/клиент вообще можно было изолированно
перенастроить портом/адресом бэкенда — без них выделенные порты выше не заработают:

- `packages/server/src/index.ts` — `openSqlite(process.env.DB_PATH)` вместо жёстко
  заданного `'kupi.db'` (функция `openSqlite` уже принимает путь параметром, меняется
  только вызов).
- `packages/client/vite.config.ts` — цель прокси `/api` берётся из
  `process.env.VITE_API_PROXY_TARGET ?? 'http://localhost:3000'` вместо хардкода
  `http://localhost:3000`.

Плюс одна точечная accessibility-правка, попутно дающая тестам стабильный локатор без
`data-testid`:

- `packages/client/src/entities/item/ui/ItemRow.tsx` — кнопка-шеврон, разворачивающая
  редактор товара, сейчас единственный элемент строки без доступного имени
  (icon-only `ActionIcon` без `aria-label`). Добавляется
  `aria-label="Редактировать {item.name}"`.

Остальная разметка приложения уже полностью доступна через `getByRole`/`getByText`
(чекбоксы связаны `label`/`htmlFor`, кнопки редактора количества имеют `aria-label`,
пункты меню и модалок — видимый текст) — новых `data-testid` не требуется.

### Изоляция тестов

Без сброса БД между тестами. Аутентификация анонимная по cookie (`kupi_dt`): каждый
Playwright-тест по умолчанию получает свежий `BrowserContext` без cookies → первый запрос
получает `401` → клиент сам вызывает `POST /api/accounts`, создавая новый аккаунт.
Тесты изолированы друг от друга бесплатно, за счёт штатного поведения Playwright — не
нужен ни сброс БД, ни ручная очистка данных между тестами.

Сценарии с двумя устройствами/аккаунтами (шеринг, линковка устройства, сверка sync)
открывают второй `browser.newContext()` внутри теста — это и есть «второе устройство».

### Структура тестового пакета

Плоские spec-файлы + тонкий набор helper-функций (без Page Object Model — экран один,
классы на каждый виджет были бы лишней абстракцией):

```
packages/e2e/
  package.json
  playwright.config.ts
  tests/
    helpers/
      actions.ts       # addItem, toggleItem, openEditor, setQuantity, pickCategory,
                        # deleteItem, openListMenu, ...
    list-crud.spec.ts
    multi-list.spec.ts
    sharing.spec.ts
    device-link.spec.ts
    offline-sync.spec.ts
```

### Покрытие сценариев

- **`list-crud.spec.ts`** — бутстрап нового устройства (первый заход создаёт аккаунт и
  дефолтный список), добавление товара (+ автодополнение по частоте использования),
  чек/анчек, разворот редактора (степпер количества, категория-чип), удаление товара.
- **`multi-list.spec.ts`** — создание нового списка, переключение между списками,
  переименование, удаление/выход (включая fallback-список при опустении).
- **`sharing.spec.ts`** — инвайт-код (8 символов): контекст A создаёт список и инвайт,
  контекст B вводит код и видит тот же список с синхронизированными товарами;
  невалидный код (короче/длиннее ожидаемого) — клиентский отказ без сетевого запроса,
  красный тост; корректный по длине, но несуществующий код — `400` с сервера, тот же
  тост.
- **`device-link.spec.ts`** — линк-код (6 символов): контекст B вводит код, видит
  предупреждающую модалку («заменит аккаунт этого устройства»), после подтверждения
  аккаунт/списки/товары полностью заменяются одним запросом (`POST /api/link`).
- **`offline-sync.spec.ts`** — `context.setOffline(true)`, локальные изменения видны
  сразу (оптимистичный патч), очередь копится; `setOffline(false)` → фоновый флаш;
  сверка через второй `browser.newContext()` того же аккаунта (переиспользуя cookie
  storageState) подтверждает, что изменения долетели до сервера.

### Скрипты

`packages/e2e/package.json`: `"test": "playwright test"`.
Корневой `package.json`: `"test:e2e": "pnpm --filter @kupi/e2e test"` — отдельно от
`pnpm test`, не блокирует быстрый unit-прогон. В CI (если появится) — отдельный шаг.

## Не в скоупе

- Кросс-браузерность (WebKit/Firefox) — только Chromium, проект для себя/близких.
- Включение в `pnpm test`/CI — по решению пользователя, отдельный `pnpm test:e2e`.
- Visual regression, a11y-аудит сверх точечной правки выше.
- Recovery для «осиротевшего» устройства после линковки — сценарий не покрыт, т.к. сама
  фича не реализована (см. `docs/backend-known-issues.md`).
