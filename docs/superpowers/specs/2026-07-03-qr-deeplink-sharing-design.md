# Дизайн: QR-коды и диплинки для шеринга списков и линковки устройств

Дата: 2026-07-03
Статус: согласован, готов к плану реализации

## Краткое описание

`docs/superpowers/specs/2026-07-02-redeem-code-design.md` и
`docs/superpowers/specs/2026-07-03-list-header-menu-redesign-design.md` оба
зафиксировали QR/диплинки как «отдельный будущий спек» — сейчас код
приглашения в список (8 симв., `POST /lists/join`) и код линковки устройства
(6 симв., `POST /link`) передаются только вручную («Копировать» копирует
голый код, получатель идёт в меню и вставляет его в текстовое поле).

Эта задача добавляет генерирующей стороне QR-картинку и кнопку «Поделиться»,
а принимающей — обработку диплинка вида `?listCode=...`/`?deviceCode=...`,
чтобы переход по ссылке или скан QR открывал сразу нужную модалку
подтверждения. Бэкенд не меняется — оба эндпоинта уже принимают голый код,
диплинк лишь несёт тот же код в URL.

Оба существующих флоу (приглашение в список и линковка устройства) получают
одинаковую инфраструктуру за один заход, а не по отдельности — они
симметричны (код + модалка + подтверждение) и делят один и тот же общий
компонент.

## Диплинк-контракт

Раздельные query-параметры, без угадывания типа кода по длине (та же
философия, что уже применили к `list-switcher`/`account-menu` в редизайне
шапки — развели по домену):

```
https://<origin>/?listCode=XXXXXXXX   — инвайт в список (8 симв.)
https://<origin>/?deviceCode=XXXXXX   — линковка устройства (6 симв.)
```

`<origin>` — `window.location.origin` во время генерации ссылки, не
захардкоженный домен (проект пока не задаёт `API_BASE_URL`/production-домен,
см. `shared/config/env.ts`) — работает одинаково в dev (`localhost:5173`) и
после деплоя.

`shared/lib/deep-link.ts` (новый файл):

- `parseDeepLink(search: string): { kind: 'list' | 'device'; code: string } | null` —
  чистая функция: читает `listCode`/`deviceCode` из `URLSearchParams(search)`,
  пустая строка или отсутствие обоих — `null`. Если оба параметра почему-то
  присутствуют одновременно (не должно происходить при обычном
  использовании) — приоритет у `listCode`. Единственная тестируемая логика в
  задаче, покрывается vitest-юнит-тестом (`deep-link.test.ts`), по аналогии с
  уже существующим `sync-status.test.ts`.
- `buildDeepLink(kind: 'list' | 'device', code: string): string` — собирает
  URL из `window.location.origin` + нужный параметр. Тривиальная
  конкатенация, без отдельного теста.

Чтение URL при загрузке приложения — в `app/App.tsx`, тем же `useRef`-guard'ом,
что уже защищает bootstrap-эффект от двойного вызова в React StrictMode:
один раз вызывает `parseDeepLink(window.location.search)`, затем
`window.history.replaceState(null, '', window.location.pathname)` — сбрасывает
URL целиком (других query-параметров в приложении сейчас нет, поэтому нет
нужды выборочно вырезать один параметр и собирать остальные обратно).

## Общий компонент: `shared/ui/CodeShareModal.tsx`

Обе существующие модалки с кодом (`ListSwitcher`'s «Пригласить в список»,
`AccountMenu`'s «Подключить это устройство») сейчас — почти идентичный JSX:
заголовок, код текстом по центру, кнопка «Копировать». Выносится в один
компонент — первая настоящая композиция в `shared/ui` (сейчас там только
реэкспорты Mantine/иконок), оправдано тем, что это дублирование в двух
местах с одинаковым поведением, и будущие правки (например, ещё один способ
шеринга) должны попадать в оба сразу, а не рассинхронизироваться.

```ts
type Props = {
  opened: boolean;
  onClose: () => void;
  title: string;
  url: string;
  code: string;
};
```

Рендерит:

1. QR-картинка сверху (раскладка «QR сверху, код снизу», выбрана в ходе
   визуального сравнения трёх вариантов) — `<img src={dataUrl}>`, где
   `dataUrl` получен через `QRCode.toDataURL(url)` (пакет `qrcode`) в
   `useEffect`, ключ эффекта — `url`.
2. Код текстом моноширинным шрифтом под QR (как сейчас) — запасной вариант
   для случаев, когда код надо продиктовать голосом, а не сканировать.
3. Кнопка «Копировать» — копирует **`url`** (полную ссылку), не голый код
   (изменение поведения: получатель в чате просто тапает по ссылке, не идёт
   вручную искать «Ввести код»).
4. Кнопка «Поделиться» — `navigator.share({ url })`, рендерится только если
   `typeof navigator.share === 'function'` (десктоп-браузеры её не имеют —
   тогда остаётся только «Копировать», без дополнительного fallback-кода).

Новая зависимость: `qrcode` + `@types/qrcode` (dev) в
`packages/client/package.json`. Библиотека, а не сторонний QR-generator API —
код приглашения/линковки не должен уходить на сторонний сервис.

### Изменения в существующих хуках/компонентах

- **`features/list-switcher/model/useListSwitcher.ts`** — `openInvite`
  дополнительно считает `url = buildDeepLink('list', code)`, кладёт в
  состояние модалки вместе с `code`.
- **`features/list-switcher/ui/ListSwitcher.tsx`** — инвайт-модалка
  заменяется на `<CodeShareModal>`.
- **`features/account-menu/model/useAccountMenu.ts`** — аналогично,
  `openLinkDevice` считает `url = buildDeepLink('device', code)`.
- **`features/account-menu/ui/AccountMenu.tsx`** — модалка кода заменяется на

  удаляется — он был зарезервирован именно под эту задачу, теперь QR встроен
  прямо в «Подключить это устройство», отдельная точка входа не нужна.

## Приём диплинка

Никаких новых модалок подтверждения — переиспользуются существующие,
диплинк лишь предзаполняет код и пропускает шаг ручного ввода. Тап по
кнопке в уже существующей модалке — и есть явное подтверждение
mutating-действия (соединение со списком / замена аккаунта устройства).

### `?listCode=` → присоединение к списку

1. `App.tsx` после разбора URL кладёt `{ kind: 'list', code }` в состояние,
   прокидывает вниз транзитом (`ListScreenPage` → `ListScreen` →
   `ListSwitcher`, тем же способом, каким уже идут `onSwitchList`/
   `onListsChanged`) как проп `initialCode?: string`.
2. `useListSwitcher` при непустом `initialCode` на маунте выполняет то же,
   что сейчас делает ручной `openCode()`, но сразу с предзаполненным
   `codeValue` — модалка «Присоединиться по коду списка» открывается с уже
   готовым кодом. Пользователь жмёт «Продолжить» → существующий `submitCode`
   (`joinList` → `onListsChanged`).
3. Повторный переход по уже использованной ссылке (пользователь уже
   участник) — не ошибка: `insertListMember` на сервере уже идемпотентен
   (`onConflict(oc) => oc.doNothing()`), `join` просто возвращает список
   повторно.
4. `400 invalid_code` (просроченный/несуществующий инвайт) — существующий
   тост, модалка с кодом остаётся открытой, как и при ручном вводе.

### `?deviceCode=` → линковка устройства

1. Аналогично прокидывается как `initialCode?: string` в `AccountMenu`.
2. `useAccountMenu` при непустом `initialCode` на маунте пропускает шаг
   ввода кода целиком и сразу выставляет `pendingLinkCode = initialCode` —
   сразу открывается предупреждающая модалка «Это заменит аккаунт этого
   устройства...» (текст без изменений). Подтверждение — существующая кнопка
   «Подключить» → `confirmLinkDevice` → `redeemLinkCode` →
   `onAccountLinked(bootstrap)`.
3. `400 invalid_code` — обрабатывается уже существующим `catch` в
   `confirmLinkDevice` без изменений: тост, `pendingLinkCode` сбрасывается,
   `deviceCodeOpen` открывается заново (с пустым `deviceCodeValue`, раз
   диплинк-код в это поле никогда не попадал) — тот же путь, что и при
   ошибке в ручном флоу, просто на пустом инпуте вместо введённого вручную.

### Сброс после обработки

Оба хука после того, как забрали `initialCode` (успешно или с ошибкой),
вызывают проброшенный сверху колбэк `onDeepLinkConsumed()`, который
обнуляет state диплинка в `App.tsx` — чтобы повторные рендеры (переключение
списка, ререндер меню) не открывали модалку снова.

### Порядок относительно bootstrap

Диплинк обрабатывается уже после того, как `lists`/`categories` загружены
(включая ветку «новое устройство → `POST /accounts`» — она и так должна
отработать раньше, `ListSwitcher`/`AccountMenu` монтируются только когда
`lists` не пусты). Специальной синхронизации не требуется — `initialCode`
приходит в уже смонтированные компоненты как обычный проп.

Особый случай: `?deviceCode=` на свежем устройстве без cookie. Bootstrap
успевает создать «одноразовый» аккаунт с дефолтным пустым списком (401 →
`POST /accounts`) до того, как отработает диплинк — тот аккаунт тут же
заменяется редимпшеном линк-кода. Один лишний аккаунт в БД на каждый такой
переход — принятый компромисс, тот же класс, что уже описан в
`docs/backend-known-issues.md` для ручного ввода кода устройства (recovery
осиротевшего аккаунта не реализован).

## Осознанно вне рамок

- **Свой in-app QR-сканер** (camera API, библиотека декодирования) —
  расчёт на штатную камеру телефона: QR кодирует полный URL, камера сама
  предлагает открыть ссылку. Не нужен ни новый permission, ни зависимость.
- **Recovery осиротевшего аккаунта** после диплинка `?deviceCode=` на
  свежем устройстве — тот же документированный gap, что и у ручного ввода
  кода (`docs/backend-known-issues.md`), не входит в эту задачу.
- **Превью названия списка** в модалке подтверждения join по диплинку —
  потребовало бы нового `GET`-эндпоинта, резолвящего код в имя списка без
  вступления. Модалка показывает только код, как и сейчас при ручном вводе.
- **Кастомный production-домен** для диплинков — используется
  `window.location.origin` в момент генерации, без отдельной конфигурации;
  актуально пересмотреть, когда клиент и сервер будут задеплоены раздельно
  (тот же пункт уже отмечен в `shared/config/env.ts` про `API_BASE_URL`).

## Тестирование

Проектная политика (`CLAUDE.md`): клиентский vitest покрывает только чистую
логику. Единственная такая логика здесь — `parseDeepLink()`, покрывается
юнит-тестом. `buildDeepLink()` — тривиальная конкатенация, без теста.
`CodeShareModal`, обработка диплинка в хуках, QR-рендеринг — необследуемо в
текущей конфигурации (нет React Testing Library), как и остальные
UI-фичи проекта.
