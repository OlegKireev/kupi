# Дизайн: приём инвайт-/линк-кода (шеринг списков между устройствами и аккаунтами)

Дата: 2026-07-02
Статус: согласован, готов к плану реализации

## Краткое описание

`docs/superpowers/specs/2026-07-01-list-header-menu-design.md` реализовал
только генерирующую сторону шеринга — «Пригласить» (`POST /lists/:id/invites`)
и «Подключить устройство» (`POST /link-codes`), обе показывают код в модалке
с кнопкой «Копировать». Приёмная сторона осталась не построена и
зафиксирована как отдельная будущая задача.

Бэкенд для приёма уже полностью готов и не требует изменений:

- `POST /lists/join` (`{ code }`) — резолвит инвайт-код (`newCode(8)`, TTL 7
  дней), добавляет вызывающий аккаунт участником списка, возвращает `List`.
- `POST /link` (`{ code }`, публичный роут) — резолвит линк-код (`newCode(6)`,
  TTL 10 минут), создаёт новое устройство на том же аккаунте, ставит
  `kupi_dt` cookie, возвращает полный `Bootstrap` (`{ account, lists,
  categories }`).

Эта задача строит только клиентский UI для ввода кода и разводки между
двумя эндпоинтами.

## Определение типа кода

Коды разной длины и не пересекаются по формату: инвайт-код списка — 8
символов, линк-код устройства — 6. Один инпут «Ввести код», тип определяется
на клиенте по длине введённой строки — без похода в сеть на заведомо
невалидную длину и без необходимости спрашивать пользователя, какой у него
код.

```
codeKind(code: string): 'list' | 'device' | 'invalid'
  length === 8 → 'list'
  length === 6 → 'device'
  иначе        → 'invalid'
```

// ponytail: длины (6/8) — деталь текущей реализации `newCode()` на сервере
(`lists/routes.ts`, `link/routes.ts`), не контракт. Если когда-то изменятся —
поправить здесь же.

## Поток данных

### Присоединение к списку (код длины 8)

1. Пользователь вводит код → `joinList(code)` → `POST /lists/join`.
2. Успех → закрываем модалку, вызываем существующий `onListsChanged(joinedList.id)`
   — тот же колбэк, что уже используется после создания/переименования/удаления
   списка (просто `GET /lists` + переключение на нужный id).
3. `400 invalid_code` → уведомление об ошибке, модалка остаётся открытой.

### Подключение устройства (код длины 6)

Редимпшн линк-кода необратимо меняет cookie этого браузера/устройства на
другой аккаунт — если на устройстве уже был свой аккаунт со своими списками,
они станут для него недоступны (recovery не реализован — известный
задокументированный gap). У любого аккаунта всегда есть хотя бы один список
(автосоздание пустого при онбординге/после последнего удаления-выхода), так
что специальной проверки «есть ли что терять» не нужно — предупреждение
показывается всегда перед этим действием.

1. Пользователь вводит код → клиент определяет тип по длине → показывает
   предупреждающую модалку («Это заменит аккаунт этого устройства. Текущие
   списки станут недоступны с него. Продолжить?»).
2. Подтверждение → `redeemLinkCode(code)` → `POST /link`.
3. Успех → сервер уже вернул полный `Bootstrap` в ответе — второй запрос не
   нужен. Вызываем новый колбэк `onAccountLinked(bootstrap)`, который в
   `App.tsx` целиком заменяет `lists`/`categories`/`activeListId` данными из
   ответа.
4. `400 invalid_code` → уведомление об ошибке, модалка ввода кода остаётся
   открытой (предупреждающая модалка к этому моменту уже закрыта).

### Невалидная длина

Ни один из двух паттернов не подошёл → уведомление об ошибке сразу, без
сетевого запроса.

## Клиент — новые/изменённые файлы

- **`entities/list/api/list-api.ts`** (+ `index.ts`) — новые функции рядом с
  уже существующими `createAccount`/`createInvite`:
  - `joinList(code: string): Promise<List>` → `POST /lists/join`
  - `redeemLinkCode(code: string): Promise<Bootstrap>` → `POST /link`
- **`features/list-switcher/model/code-kind.ts`** (+ `.test.ts`) —
  `codeKind()`, единственная чистая логика в этой задаче, покрывается
  юнит-тестом по аналогии с `list-menu/model/sync-status.test.ts`.
- **`features/list-switcher/model/useListSwitcher.ts`** — новый хук,
  вытягивает состояние из `ListSwitcher.tsx` (по образцу
  `list-menu/model/useListMenu.ts`): существующее состояние модалки «Новый
  список» + новое состояние ввода кода + состояние предупреждающей модалки
  для линк-кода.
- **`features/list-switcher/ui/ListSwitcher.tsx`** — переезжает на
  `useListSwitcher`, добавляет пункт меню «Ввести код» и две модалки (ввод
  кода; подтверждение подключения устройства).
- **`app/App.tsx`** — новый обработчик:
  ```ts
  const onAccountLinked = (bootstrap: Bootstrap): void => {
    setLists(bootstrap.lists);
    setActiveListId(bootstrap.lists[0]?.id ?? null);
    setCategories(bootstrap.categories);
  };
  ```
  Прокидывается вниз как новый проп через `ListScreenPage` → `ListScreen` →
  `ListSwitcher` (транзитом, тем же способом, каким уже идут
  `onSwitchList`/`onListsChanged`).
- **`pages/list-screen/ui/ListScreenPage.tsx`**,
  **`widgets/list-screen/ui/ListScreen.tsx`** — пробрасывают новый проп
  транзитом, без собственной логики.

## Уведомления об ошибках

Новая зависимость: `@mantine/notifications` (пользовательское решение —
альтернатива, инлайн-текст в модалке, была бы дешевле, но `notifications` даёт
первый переиспользуемый паттерн показа ошибок в приложении, которого сейчас
нет нигде).

- `packages/client/package.json` — добавить зависимость.
- `main.tsx` — смонтировать `<Notifications />` рядом с `MantineProvider`,
  импортировать `@mantine/notifications/styles.css`.
- Используется для: `400 invalid_code` от `joinList`/`redeemLinkCode`,
  сетевых ошибок (не `ApiError` — паттерн `err instanceof ApiError` уже
  применяется в `App.tsx`/`useItemSync.ts`), и невалидной длины кода.

## Тестирование

Проектная политика (`CLAUDE.md`): клиентский vitest покрывает только чистую
логику, без React Testing Library. Единственная чистая логика в этой задаче —
`codeKind()`, покрывается юнит-тестом. Остальное (сетевые вызовы, UI,
модалки) — необследуемо в текущей конфигурации, как и остальные фичи
(`useListMenu`, `useItemSync` и т.д.).

## Осознанно вне рамок

- Диплинки/URL с кодом (`?code=...`) и авто-открытие модалки при переходе по
  ссылке — сейчас коды передаются только вручную (как и раньше — «Копировать»
  копирует голый код, не ссылку). Добавить, когда появится реальный сценарий
  шеринга через мессенджер/QR.
- QR-коды для линковки — упомянуто в `docs/backend-known-issues.md` как
  отдельный будущий пункт.
- Recovery/восстановление доступа к «осиротевшему» аккаунту после
  редимпшна линк-кода — известный задокументированный gap
  (`docs/backend-known-issues.md`), не входит в эту задачу.
