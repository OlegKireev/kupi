# Дизайн: реорганизация меню шапки списка

Дата: 2026-07-03
Статус: согласован, готов к плану реализации

## Краткое описание

Текущая шапка (`docs/superpowers/specs/2026-07-01-list-header-menu-design.md`)
смешивает в одном пункте меню («⋮», `features/list-menu`) два разных домена:
действия над **текущим списком** (пригласить, участники, переименовать,
удалить) и действия над **аккаунтом/устройством** (подключить устройство).
Пользовательская обратная связь:

- иконка «⋮» не читается как «меню именно этого списка» — выглядит как
  общее меню приложения;
- «Переименовать список» — лишний пункт меню, раз есть кликабельный
  заголовок;
- «Пригласить» и «Подключить устройство» рядом в одном меню путают —
  обе открывают одинаковую модалку с кодом, разница в назначении кода
  неочевидна;
- «Ввести код» в меню переключения списков обрабатывает и код списка
  (8 симв.), и код устройства (6 симв.) через угадывание по длине — код
  устройства логически не относится к списку вообще.

Решение: развести на две точки входа по домену — заголовок списка (всё
про списки) и новая иконка аккаунта (всё про устройство). Инлайн-
редактирование заголовка рассмотрели и отклонили: первый тап по
заголовку должен предсказуемо открывать меню, а не рисковать попасть в
режим редактирования — переименование остаётся пунктом меню с уже
существующей модалкой.

Сознательно вне рамок этой задачи:

- QR-флоу для подключения устройства — только резервируется пункт меню
  (disabled, с пояснением «скоро»), генерация/сканирование QR не
  реализуются. Отдельный будущий спек.
- Строка статуса синка остаётся как есть (уже реализована), в
  расположении не меняется — рассмотрели вынос из меню, отклонили как
  избыточное усложнение шапки ради второстепенной информации.

## Клиент

### Шапка (`widgets/list-screen/ui/ListScreen.tsx`)

Было: `<ListSwitcher .../> <ListMenu .../>` рядом в одной `Group`.
Станет: `<ListSwitcher .../> <AccountMenu .../>` — `ListMenu` как
отдельный компонент исчезает, его содержимое поглощает `ListSwitcher`.

### `features/list-switcher` (расширяется)

Один `Menu`, открывается тапом по заголовку (текст + `CaretDown`, как и
сейчас — единая кликабельная зона, никакого отдельного инлайн-режима).
Пункты сверху вниз:

1. **Секция текущего списка** (новое, поглощено из `list-menu`):
   - статус синка — неинтерактивный `Menu.Label` (`getSyncStatusText`,
     без изменений)
   - «Пригласить в список» → `createInvite`, код-модалка с копированием
   - «Участники (N)» — лениво подгружает `getMemberCount` при открытии
     меню, неинтерактивный пункт
   - «Переименовать список» → модалка с `TextInput`, предзаполненным
     текущим именем → `renameList` (без изменений логики, просто теперь
     в общем меню)
   - «Удалить/покинуть список» → confirm-модалка → `deleteList`
2. `Menu.Divider`
3. **Остальные списки пользователя** (как сейчас), пункт активного
   списка визуально подсвечен (`data-active`/фон)
4. `Menu.Divider`
5. «Новый список» → модалка с `TextInput` → `createList`
6. «Присоединиться по коду списка» → модалка с `TextInput` (только код
   списка, 8 симв.) → `joinList`. Переименовано из «Ввести код» —
   больше не принимает код устройства, угадывание типа по длине не
   нужно.

`useListSwitcher` вбирает в себя весь `useListMenu` (sync-статус,
инвайт, участники, переименование, удаление) и теряет
device-link-ветку `submitCode` (`codeKind === 'device'`/
`pendingLinkCode`/`confirmLinkDevice`/`cancelLinkDevice`) — эта логика
переезжает в `account-menu`. `submitCode` для кода списка упрощается:
сразу `joinList`, без ветвления по `codeKind`.

Пропс `onAccountLinked` у `ListSwitcher`/`useListSwitcher` убирается (не
нужен — device-линковка переехала в `account-menu`).

### `code-kind.ts` — удаляется

Вместе с `code-kind.test.ts`. Каждый инпут кода теперь однозначно
привязан к одному домену (список или устройство), угадывание типа по
длине больше не требуется.

### `features/account-menu` (новый слайс, по образцу `list-menu`)

`ActionIcon` с `UserCircleIcon` (вместо старого `DotsThreeVertical`) на
месте, где раньше был `ListMenu`. Меню:

- «Подключить это устройство» → `createLinkCode` (уже существующая
  `link-code-api.ts`, переезжает сюда из `list-menu`), код-модалка с
  копированием — как «Пригласить», просто другой код
- «Ввести код устройства» → модалка с `TextInput` (только код
  устройства, 6 симв.) → confirm-модалка с предупреждением «заменит
  аккаунт этого устройства» (текст без изменений) →
  `redeemLinkCode` → `onAccountLinked(bootstrap)`. Логика — прямой
  перенос текущей device-ветки из `useListSwitcher`
  (`pendingLinkCode`/`confirmLinkDevice`/`cancelLinkDevice`), только
  вход в неё теперь без угадывания по `codeKind`
- «QR-код (скоро)» — disabled `Menu.Item` с `QrCodeIcon`, без обработчика

Файлы: `ui/AccountMenu.tsx`, `model/useAccountMenu.ts`, `index.ts`.
`onAccountLinked: (bootstrap: Bootstrap) => Promise<void>` — тот же
пропс, что был у `ListSwitcher`, переезжает сюда из
`app/App.tsx` → `ListScreen` → `AccountMenu`.

### Иконки

Новые: `UserCircleIcon`, `QrCodeIcon` (оба уже есть в
`@phosphor-icons/react`, добавляются в реэкспорт `shared/ui/index.ts`).
`DotsThreeVerticalIcon` используется только в удаляемом `ListMenu.tsx` —
убрать из реэкспорта вместе с ним.
