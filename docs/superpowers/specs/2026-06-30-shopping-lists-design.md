# Дизайн: совместные списки покупок (offline-first PWA)

Дата: 2026-06-30
Статус: согласован, готов к плану реализации

## Краткое описание

Веб-приложение (PWA) для списков покупок с упором на:

- **Удобство** — максимально быстрое добавление/удаление товаров, автодополнение и
  сортировка предложений по популярности (частоте использования).
- **Offline-first** — в магазине без интернета список полностью работает; изменения
  копятся локально и синхронизируются с бэком при появлении сети.
- **Шеринг** — один аккаунт на нескольких своих устройствах **и** общие списки,
  редактируемые несколькими пользователями.
- **Без барьера на входе** — никакой обязательной регистрации; открыл и пользуешься.
- **Простой, прозрачный дизайн.**

Драйвер проекта — собственный сервис с нуля, которым автор пользуется сам и делится
с близкими. Монетизация вторична и в код MVP не закладывается.

## Ключевые продуктовые решения

| Решение                | Выбор                                                                    |
| ---------------------- | ------------------------------------------------------------------------ |
| Аутентификация         | Anonymous-first: молчаливый анонимный аккаунт при первом запуске         |
| Мультиустройство       | Привязка устройств по короткому коду к одному аккаунту                   |
| Совместные списки      | Инвайт-код/ссылка на список, вступление других аккаунтов                 |
| Восстановление доступа | Отложено (привязка 2-го устройства + recovery-код/VK — потом)            |
| Платформа клиента      | PWA на React; путь в сторы через Capacitor — позже, без переписывания    |
| Источник истины        | Бэк; локальная IndexedDB — производный оффлайн-кэш                       |
| Realtime               | Нет; синк при открытии + фоновый pull. WebSocket/SSE — потом, если нужно |

## Архитектура

```
[React PWA]  --HTTP/REST-->  [Fastify]  -->  [SQLite (один файл)]
  IndexedDB (кэш + очередь pending changes)
```

Валидация — zod: схемы лежат в `shared` (один источник правды), сервер через
`fastify-type-provider-zod` валидирует вход (body/params/query) и выводит типы оттуда же.

Один лёгкий Node-процесс, БД — файл SQLite. Никаких внешних сервисов (SMTP, OAuth,
очередей). «Нетребовательность к железу» обеспечивается отсутствием отдельной СУБД
и realtime-инфраструктуры.

## Аутентификация и доступ

### Anonymous-first

- Первый запуск: клиент шлёт `POST /accounts` → сервер генерит `account_id` и
  `device_token` (32 байта CSPRNG), пишет в БД с `UNIQUE` на токен (коллизия
  конструктивно исключена), возвращает токен в cookie.
- Токен живёт в **`HttpOnly; Secure; SameSite=Lax; Path=/` cookie** `kupi_dt` (через
  `@fastify/cookie`), выставленной сервером (переживает Safari ITP 7-day cap, не доступен
  XSS). Для QR/кодов токен из JS читать не нужно. `POST /accounts` сразу создаёт дефолтный
  список «Мои покупки» и возвращает bootstrap (lists + categories). Auth — `onRequest`-хук:
  cookie → device → `request.accountId`; публичны только `POST /accounts`, `POST /link`,
  `GET /health`.
- Все запросы авторизуются по cookie; сервер проверяет токен по БД (opaque, без JWT,
  отзываемый).

### Продление куки (sliding expiration)

`Max-Age` cookie ограничен 400 сутками (Chrome cap). Сервер на каждом
authenticated-запросе переустанавливает `Set-Cookie` с новым `Max-Age=400d`. Пока
пользователь заходит хотя бы раз в <400 дней — доступ не теряется. Сам `device_token`
в БД не протухает; ротацию токена не делаем.

Продление куки — это только лишний `Set-Cookie`-заголовок в ответе, **без записи в БД**:
токен уже прочитан при проверке доступа, новый `Max-Age` ничего не обновляет в БД. Нагрузки
на БД per-request продление не создаёт.

Продление меняет **только TTL куки, не её значение** — `device_token` остаётся тем же.
У каждого устройства свой токен (своя запись `devices` с общим `account_id`), поэтому
продление куки на одном устройстве не трогает куки других. Одновременный вход нескольких
устройств одного аккаунта работает без вытеснения: сервер по каждому токену находит свой
`account_id`.

### Привязка своих устройств

1. Устройство A: `POST /link-codes` → короткий код (6 символов, TTL ~10 мин).
2. Устройство B: `POST /link` с кодом → сервер создаёт новую запись `devices` с тем же
   `account_id`, выдаёт B собственный токен в cookie.
3. Теперь A и B видят одни и те же данные.

QR в MVP не делаем — код вводится вручную. QR-сканер — улучшение поверх.

### Совместные списки

1. Владелец: `POST /lists/:id/invites` → инвайт-код/ссылка (TTL настраиваемый).
2. Другой пользователь (со своим анонимным аккаунтом): `POST /lists/join` с кодом →
   запись в `list_members`.
3. Доступ к списку и его товарам строго проверяется по членству (изоляция: чужой
   аккаунт не видит и не меняет список).

### Риски доступа (осознанно приняты)

Потеря всех привязанных устройств = потеря аккаунта (нет recovery). Для списков
покупок ценность данных низкая — приемлемо. Митигации на будущее: подсказка
«подключи второе устройство», recovery-код, опциональная VK-привязка. В MVP не входят.

## Синхронизация (offline-first + optimistic UI)

### Принципы

- **Источник истины — сервер.** IndexedDB на клиенте — кэш + очередь несинхронизированных
  изменений (pending).
- **Optimistic UI:** действие пользователя сразу применяется к локальному состоянию и
  UI, попадает в очередь pending. Синхронизация идёт **в фоне и никогда не блокирует
  действие**.
- **Когда синкаем:** при открытии приложения; в фоне перед/после изменений; по событию
  `online`.
- **Гранулярность — товар (item), не список.** Операции над списком покупок почти
  коммутативны, поэтому конфликты редки и дёшевы.

### Версионирование

- На каждый список — монотонный счётчик `seq`. Любое изменение товара инкрементит
  `list.seq` и проставляет `item.version = seq`.
- Дельта-pull: клиент хранит `last_seen_seq` по списку и запрашивает всё с
  `version > last_seen_seq`.
- Wall-clock устройства **не используется** как арбитр. Арбитр порядка между
  устройствами — сервер (серверный `seq`). Локальное время — только для упорядочивания
  собственных оффлайн-правил до отправки.

### Протокол синка (REST)

`POST /lists/:id/sync` (auth + проверка членства):

- Запрос: `{ lastSeenSeq, changes: [{ itemId, clientOpId, op: "upsert"|"delete",
fields: { name?, quantity?, categoryId?, checked? } }] }`. Каждое change несёт
  `clientOpId` (идемпотентность), целевой `itemId` (генерится клиентом), тип операции и
  **только изменённые поля** (патч). Клиент версию не шлёт — арбитр порядка серверный.
- Сервер (в транзакции, по порядку): `INSERT OR IGNORE applied_ops(clientOpId)` →
  если уже было, пропуск; нет товара → создать с `version = ++list.seq`; есть и
  `deleted=1`, а op≠delete → пропуск (remove-wins); иначе применить переданные поля,
  `version = ++list.seq`. Возвращает `{ seq, items: [...всё с version > lastSeenSeq,
включая tombstones...] }`.

### Правила разрешения конфликтов (silent merge, без модалок)

- **Разные товары** → не конфликт, применяются оба.
- **Один товар, разные поля** (один правит `quantity`, другой `category`) → обе правки
  выживают: changes несут только изменённые поля (патч) и применяются по-колоночно.
- **Один товар, одно поле** → побеждает change, дошедший до сервера последним (whole-item
  LWW по серверному `seq`; wall-clock не арбитр). Одна колонка `version` нужна только как
  курсор дельта-pull — per-field версий нет.
- **check vs remove** → remove (tombstone) побеждает; удалённое не воскресает правкой.
- **Удаление = tombstone** (мягкое, с версией). В MVP tombstones и `applied_ops` не
  чистятся — sweep по TTL добавим, когда таблицы вырастут (`// ponytail`).
- **Двойное добавление одного названия оффлайн** → в MVP принимаем оба (дубликат не
  критичен). Дедуп по нормализованному имени — потом.
- Конфликты пользователю не показываются.

## Модель данных (SQLite)

- `accounts` — `id`, `created_at`. Анонимный.
- `devices` — `id`, `account_id`, `token` (UNIQUE), `created_at`.
- `link_codes` — `code`, `account_id`, `expires_at`.
- `lists` — `id`, `name`, `owner_account_id`, `seq`, `created_at`.
- `list_members` — `list_id`, `account_id`, `role` (`owner`/`member`).
- `list_invites` — `code`, `list_id`, `expires_at`.
- `items` — `id`, `list_id`, `name`, `quantity`, `category_id`, `checked`, `version`
  (= `list.seq` на момент правки), `deleted` (tombstone), `updated_at`.
- `applied_ops` — `client_op_id` (PK). Идемпотентность синка; в MVP не чистится.
- `categories` — `id`, `name`, `color`, `icon`. **Пресетный общий набор** (~9: Овощи и
  фрукты, Молочное, Мясо и рыба, Бакалея, Напитки, Хлеб, Заморозка, Бытовое, Другое),
  сидится при инициализации БД; кастомные категории — потом.
- `item_frequency` — `account_id`, `normalized_name` (trim+lowercase), `count`. Инкремент
  при создании товара; питает `GET /suggestions?q=` (префикс, сортировка по `count`).

Идентификаторы — `crypto.randomUUID()` (stdlib, обе стороны); `item`/`list` id генерит
**клиент** (стабильность оптимистичных товаров оффлайн). Схема — один `schema.sql`
(`CREATE TABLE IF NOT EXISTS` + `PRAGMA user_version` на будущее), без миграционного
фреймворка.

## Клиент (PWA)

- **React + TypeScript + Vite** + `vite-plugin-pwa` (service worker, install,
  оффлайн-кэш ассетов).
- **IndexedDB** через `idb` (тонкая обёртка) — кэш данных + очередь pending changes.
- **Состояние** — встроенный React (`useReducer`/Context), без Redux.
- **Install to home screen** мягко поощряется (снимает Safari 7-day eviction);
  `navigator.storage.persist()` как доп. страховка для IndexedDB.
- UI: быстрый ввод товара с автодополнением (по популярности), отметка/удаление в один
  тап, категории с цветами/иконками, индикатор оффлайн/синка.

## Тестирование

Бэк — тесты на все критичные core-фичи (Fastify `inject` + `node:test`):

- **Вход:** создание аккаунта/устройства, выдача и валидация токена, sliding-продление.
- **Привязка своих устройств:** link-code, привязка B к аккаунту A, общий доступ к данным.
- **Шеринг списков:** инвайт-код, вступление, членство, **изоляция доступа** (чужой не
  видит/не меняет).
- **Манипуляции и синк:** CRUD товаров, дельта-pull по `seq`, правила мерджа (per-field
  LWW, remove-vs-edit), идемпотентность по `client_op_id`.
- **Популярность:** инкремент частоты и сортировка предложений.

Клиент — без тестов на старте (добавим точечно, если логика синка на клиенте
усложнится).
