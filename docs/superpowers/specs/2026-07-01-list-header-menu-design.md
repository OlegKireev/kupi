# Дизайн: шапка списка + меню «⋮»

Дата: 2026-07-01
Статус: согласован, готов к плану реализации

## Краткое описание

Первый вертикальный срез клиента (`docs/superpowers/plans/2026-07-01-client-first-slice.md`)
реализовал только тело экрана — добавление/чек/правку/удаление товара. Шапка
(переключение списков, меню «⋮» с приглашением/участниками/подключением
устройства/переименованием/удалением) осталась нереализованной, хотя описана в
`docs/superpowers/specs/2026-07-01-client-ui-design.md`.

При сведении того UI-спека с реальным API сервера всплыли два пробела,
которые UI-спек не учёл (написан без сверки с бэкендом):

- **Число участников списка** нигде не возвращается — ни в `List`, ни отдельным
  роутом.
- **Удаление/выход из списка** — роута `DELETE /lists/:id` не существует.

Оба пробела закрываются в этой же задаче (решение пользователя), а не
выносятся отдельно.

Сознательно вне рамок этой задачи (решения зафиксированы при брейнсторминге):

- Приём инвайт-/линк-кода по ссылке или ручному вводу — экран-приёмник не
  строим, только сторона генерации кода.
- Строка статуса синка в меню («Синхронизировано только что» / «N в
  очереди») — требует offline-очереди изменений на клиенте, которой ещё нет
  (отдельный будущий спек по sync-движку). В меню эта строка сейчас не
  показывается вообще.
- Свайп для удаления товара.

## Бэкенд

### `DELETE /lists/:id` (protected, `lists/routes.ts`)

Единый роут, поведение зависит от роли вызывающего — сам пункт меню на
клиенте один и тот же («Удалить/покинуть список»), решает сервер:

- не член списка → `404 { error: 'not_found' }` (как у всех остальных
  list-роутов — не палим существование списка)
- владелец (`isOwner`) → удаляет список целиком одной транзакцией: `items`
  (по `list_id`), `list_invites` (по `list_id`), `list_members` (по
  `list_id`), затем саму строку `lists`
- участник, не владелец → удаляет только свою строку в `list_members`
  (выход, остальные данные списка не трогает)
- ответ: `204`, без тела

Каскад — вручную в транзакции: в схеме (`db/schema.ts`) внешние ключи объявлены
без `ON DELETE CASCADE`, так что таблицы нужно чистить в явном порядке.

### `GET /lists/:id/members` (protected, `lists/routes.ts`)

- не член → `404 { error: 'not_found' }`
- член → `{ count: number }` — только число, без списка участников (см.
  UI-спек: имён участников в MVP нет)

### `repository.ts` (новые функции)

- `deleteList(trx, listId)` — удаляет `items`/`list_invites`/`list_members`/
  `lists` по `listId` (используется на owner-пути внутри транзакции)
- `removeListMember(db, listId, accountId)` — удаляет одну строку
  `list_members` (member-путь)
- `countListMembers(db, listId): Promise<number>` — `count(*)` по
  `list_members` для `listId`

### `@kupi/shared`

Новый тип для ответа `GET /lists/:id/members`: `{ count: number }` — либо
инлайн zod-схема в роуте, либо маленький `MemberCountSchema` в
`packages/shared/src/index.ts` (по аналогии с остальными response-схемами).

## Клиент

### `entities/list` (расширение существующего слайса)

Новые функции в `api/list-api.ts`, экспортируются через `index.ts`:

- `renameList(id: string, name: string): Promise<List>` → `PATCH /lists/:id`
- `deleteList(id: string): Promise<void>` → `DELETE /lists/:id`
- `createList(name: string): Promise<List>` → `POST /lists`
- `getMemberCount(id: string): Promise<number>` → `GET /lists/:id/members`

### `features/list-switcher` (новый слайс)

Заголовок списка в шапке + `CaretDown` рядом. Тап открывает Mantine `Menu`:
пункт на каждый список из `lists`, выбор — переключение активного; внизу
пункт «Новый список», открывающий маленький inline-`Modal` с `TextInput` и
кнопкой создания (`createList`).

Файлы: `ui/ListSwitcher.tsx`, `index.ts`. Данные (`lists`, `activeListId`,
колбэки) — пропсами сверху, слайс не хранит список списков сам.

### `features/list-menu` (новый слайс)

Одна UX-сцена «меню ⋮ списка», по аналогии с тем, как `edit-item` бандлит
степпер+чипы+удаление в одну фичу. `ActionIcon` с `DotsThreeVertical`
открывает Mantine `Menu`:

- **Пригласить** — вызывает `POST /lists/:id/invites`, открывает `Modal` с
  кодом (крупный текст) и кнопкой «Копировать» (`Copy` icon,
  `navigator.clipboard.writeText`)
- **Участники (N)** — при открытии меню лениво подгружает `getMemberCount`,
  показывает как неинтерактивный пункт-лейбл
- **Подключить устройство** — вызывает `POST /link-codes`, тот же
  код-`Modal` с копированием, что и у приглашения
- **Переименовать список** — `Modal` с `TextInput`, предзаполненным текущим
  именем → `renameList`
- **Удалить/покинуть список** (`Trash` icon) — confirm-`Modal` с общим
  текстом (клиент не различает владельца/участника — сервер решает
  поведение сам) → `deleteList`, затем перезапрос списков на уровне `App`

Файлы: `ui/ListMenu.tsx`, `index.ts`.

### Иконки

Новая зависимость `@phosphor-icons/react` (пользовательское решение —
вместо текстовых `▾`/`⋮` из исходного UI-спека). Используемые иконки:
`CaretDown`, `DotsThreeVertical`, `Copy`, `Trash`.

### Состояние (`app/App.tsx`)

Поднимается на уровень `App`:

- `lists: List[]` вместо одного `list`
- `activeListId: string | null`
- активный список — `lists.find(l => l.id === activeListId)`

После любой мутации списка (создание/переименование/удаление-выход) —
простой `refetch` `getLists()` и обновление стейта, без ручного патчинга
(не hot path, лишняя сложность не нужна). Если после удаления/выхода
`lists` пуст — автоматически `createList('Мои покупки')`, тот же паттерн,
что при онбординге нового аккаунта.

`ListScreenPage` → `ListScreen` получают `lists`, `list` (активный),
`onSwitchList`, `onListsChanged` дополнительно к уже существующим
`categories`.

## Осознанно отложено

- Приём инвайт-/линк-кода (диплинк + ручной ввод) — отдельная задача.
- Строка статуса синка в меню — ждёт отдельного спека по sync-движку.
- Свайп для удаления товара.
- Полноценный список участников с именами (в модели данных имён нет вообще).
