# Дизайн: FSD-архитектура клиента (скелет + первый вертикальный срез)

Дата: 2026-07-01
Статус: согласован, готов к плану реализации

## Краткое описание

`packages/client` сейчас — пустой каркас (React 19 + Vite, только заглушка в
`App.tsx`/`main.tsx`, ни роутера, ни стейт-менеджера, ни тестов). Этот
документ фиксирует Feature-Sliced Design структуру для клиента и объём первого
вертикального среза: экран активного списка с товарами через прямые запросы к
уже существующему REST API сервера (без офлайн-синка/IndexedDB — это отдельный
будущий спек, см. `docs/superpowers/specs/2026-07-01-client-ui-design.md`).

UI/UX-решения (экраны, визуальный стиль, взаимодействие с товаром) уже
зафиксированы в `2026-07-01-client-ui-design.md`. Этот документ не повторяет
их, только описывает техническую/архитектурную организацию кода.

## Ключевые решения

| Область          | Решение                                                                  |
| ---------------- | ------------------------------------------------------------------------ |
| Архитектура      | FSD, все 6 слоёв: `app → pages → widgets → features → entities → shared` |
| Границы слоёв    | `steiger` (официальный FSD-линтер, не ESLint — не конфликтует с oxlint)  |
| Данные с сервера | Голый `fetch` (`shared/api/client.ts`), без TanStack Query               |
| Состояние        | `useState`, поднятое в `App.tsx`, без Context/стора — один экран         |
| Тесты            | Не настраиваем сейчас — нет логики, которую стоит покрывать              |
| Слой `pages`     | Заводим сразу (`pages/list-screen`), для единообразия на будущее         |

## Слои и правила импорта

Стандартные 6 слоёв FSD сверху вниз. Слой ниже не знает о слоях выше; импорт
только «вниз» и только через public API слайса (`index.ts`).

- **Алиасы**: переиспользуем существующий `@/*` → `./src/*` из tsconfig,
  дополнительный алиас на слой не нужен.
- **Границы слоёв**: `steiger` добавляется как devDependency в
  `packages/client`, запускается отдельной командой (например,
  `pnpm --filter @kupi/client fsd-lint`), не встроен в `pnpm lint`
  (`oxlint`) — независимая проверка.
- **Данные с сервера**: `shared/api/client.ts` — тонкая обёртка над `fetch`
  (базовый URL, `credentials: 'include'` для cookie-авторизации `kupi_dt`,
  JSON-парсинг, проброс не-2xx как ошибки). Кэширования/инвалидации нет —
  каждый компонент дергает fetch сам через хуки в features/widgets.

## Карта слайсов

### entities

- **`entities/list`** — тип `List` (реэкспорт из `@kupi/shared`), `api.ts`:
  `getLists()` (`GET /lists`), `createAccount()` (`POST /accounts`, только
  для первого запуска).
- **`entities/category`** — тип `Category`, `api.ts`: `getCategories()`
  (`GET /categories`), `ui/CategoryIcon.tsx` (эмодзи-иконка + цветная точка
  по `category.icon`/`category.color`, см. визуальный стиль в UI-спеке).
- **`entities/item`** — тип `Item`, `api.ts`: `syncItems(listId, req:
SyncRequest)` — единственная точка записи (`POST /lists/:id/sync`;
  создание/чек/правка/удаление — всё через `ItemChange` с `op: 'upsert' |
  'delete'`), `ui/ItemRow.tsx` (read-only отображение строки: чекбокс,
  зачёркивание отмеченного, иконка категории, без интерактивности разворота
  — это уровень фичи).

### features

Каждая фича вызывает `entities/item`'s `syncItems` с нужным `ItemChange`
(`clientOpId` генерируется через `crypto.randomUUID()` в
`shared/lib/ids.ts` — сервер использует его для идемпотентности).

- **`features/add-item`** — текстовый инпут + выпадающий список
  автодополнения (`GET /suggestions?q=`, живёт в этой фиче, а не в
  `entities/item`, т.к. это специфика одного сценария ввода, не общая
  для сущности «товар» операция). Выбор подсказки создаёт товар с
  категорией найденного совпадения; «Добавить как новый» — без категории.
- **`features/toggle-item`** — чек/анчек по тапу на чекбокс.
- **`features/edit-item`** — разворачивание строки по тапу (не по
  чекбоксу): степпер количества, чипы категорий, ссылка «Удалить товар».
  Один слайс на все три под-действия, т.к. они — одна UX-сцена «раскрытая
  строка», не независимые сценарии.

### widgets

- **`widgets/list-screen`** — композиция: статичный заголовок списка (без
  `▾`-меню переключения — см. «Отложено»), `features/add-item`, список
  `ItemRow` с подключёнными `features/toggle-item` и `features/edit-item`,
  пустое состояние с текстовой подсказкой (см. UI-спеку, «Онбординг»).

### pages

- **`pages/list-screen`** — тонкая обёртка над `widgets/list-screen`, без
  собственной логики.

### app

- **`app/App.tsx`** — bootstrap-логика сессии (см. «Поток данных» ниже) +
  рендер `pages/list-screen`.
- **`app/styles/globals.css`** — CSS custom properties из визуального стиля
  UI-спеки: `--color-bg: #fdfcfa`, `--color-accent: #c9764f`, `--color-row-alt:
#f7f4ef`, системный шрифтовый стек, скругления строк (~10px).

## Поток данных при загрузке

Оркестрация нескольких entities (list + category + неявно account) — не
бизнес-логика одной сущности, поэтому живёт в `app`, а не в
`entities/list`:

1. `GET /lists` + `GET /categories` параллельно (`Promise.all`) — сработает
   для уже знакомого устройства (cookie `kupi_dt` есть).
2. Если запрос вернул `401` (новое устройство, нет cookie) → `POST
/accounts` — создаёт account+device+дефолтный список одной транзакцией на
   сервере, ставит cookie, возвращает `Bootstrap` (`{ account, lists,
categories }`) целиком одним запросом — второй `GET /categories` в этом
   случае не нужен.
3. Результат (`lists[0]`, `categories`) — `useState` в `App.tsx`,
   пробрасывается пропсами вниз до `pages/list-screen` → `widgets/list-screen`.
   Без Context/глобального стора: на один экран это лишняя инфраструктура.

## Осознанно отложено

Не входит в первый вертикальный срез — отдельные будущие срезы/планы:

- **Шапка**: `▾`-переключение списков и `⋮`-меню (инвайт в список,
  участники, привязка устройства, переименование, удаление/выход, статус
  синка). Первый срез рендерит только статичный заголовок. Кластер из 5+
  независимых фич, не связанных с «отрисовать список товаров».
- **Приём инвайта / привязки устройства по deep-ссылке** (`window.location`
  парсинг из UI-спеки) — та же группа, тот же будущий срез.
- **Свайп для удаления товара** — альтернативный способ, отложен; основной
  путь удаления через `features/edit-item` достаточен для первого среза, не
  тянем зависимость под жест.
- **IndexedDB / офлайн-очередь pending changes / retry** — по UI-спеке это
  отдельный будущий спек клиентского sync-движка; `syncItems` сейчас просто
  бьёт в сеть напрямую, без локального кэша.
- **Vitest / тестовая инфраструктура** — добавим, когда появится первая
  логика, которую стоит покрывать (например, sync-движок), а не сейчас ради
  самого факта наличия тестов.

Входит в первый срез (несмотря на маленький объём — почти бесплатно рядом с
основной функциональностью): онбординг-подсказка в пустом списке
(текст под полем ввода при `items.length === 0`).
