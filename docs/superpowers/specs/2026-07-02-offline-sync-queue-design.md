# Offline-очередь и sync-статус — дизайн

## Проблема

Сейчас каждая мутация (`toggle`/`add`/`edit`/`delete`) синхронно шлёт
`POST /lists/:id/sync` и ждёт ответ. Если сети нет — запрос падает
необработанным, UI не обновляется, ничего не сохраняется локально. При
перезагрузке страницы список пуст, пока не пройдёт сеть (`ListScreen`
обнуляет `items` на маунте и ждёт `syncItems`). Три feature-хука
(`useToggleItem`, `useAddItem`, `useEditItem`) независимо друг от друга
строят `ItemChange` и напрямую вызывают `syncItems`/`onSynced` — синк-логика
размазана.

Отложено ещё в спеке `2026-07-01-list-header-menu-design.md`: «строка
статуса синка в меню — ждёт отдельного спека по sync-движку». Это он.

## Объём фичи

1. Очередь несинканных мутаций — действия офлайн не теряются, досылаются при
   восстановлении сети.
2. Офлайн-чтение — список виден при открытии приложения без сети вообще, из
   локального кеша (не только «действие поставлено в очередь», но и
   «список открывается и что-то показывает»).
3. Sync-статус в `features/list-menu`.

## Хранилище

`localStorage`, без новых зависимостей (`IndexedDB` не нужен на объёме
данных списка покупок; синхронный API `localStorage` проще стыкуется с
React-стейтом, чем асинхронный `IndexedDB`).

- `kupi:list:<listId>` →
  ```ts
  type ListCache = {
    items: Item[];
    lastSeenSeq: number;
    queue: QueuedChange[];
  };
  type QueuedChange = {
    change: ItemChange;
    attempts: number;
    failed: boolean;
  };
  ```
- `kupi:bootstrap` → `{ account: Account, lists: List[], categories: Category[] }`
  — снимок того, что сейчас отдают `POST /accounts` /
  `GET /lists`+`GET /categories`, нужен для холодного старта офлайн.

Оба кеша пишутся при каждом изменении соответствующего стейта, читаются один
раз при маунте.

## Централизованный синк-хук

Новый `entities/item/model/useItemSync.ts` заменяет прямые вызовы
`syncItems`/`onSynced` в фичах:

```ts
function useItemSync(listId: string): {
  items: Item[];
  pendingCount: number;
  failedCount: number;
  applyChange: (change: ItemChange) => void;
};
```

При маунте читает `kupi:list:<listId>` из `localStorage` — `items`
отрисовываются немедленно, без сети (stale-while-revalidate: если данные
устарели, флаш в фоне подтянет актуальные — специальной логики «сравнить и
обновить» не нужно, см. ниже). Затем пытается флашнуть очередь в фоне.

`applyChange(change)`:

1. Оптимистично патчит `items` локально — новая чистая функция
   `applyChangeLocally(items, change)` рядом с существующим `mergeItems` в
   `entities/item`. Простой upsert/delete по `itemId`, без LWW-логики
   сервера (на одном устройстве правки всегда «последние»).
2. Добавляет `{ change, attempts: 0, failed: false }` в очередь, пишет
   `items`+`queue` в `localStorage`.
3. Если `navigator.onLine` — пытается сразу флашнуть.

`useToggleItem`/`useAddItem`/`useEditItem` перестают знать про `syncItems`:
строят `ItemChange` как сейчас, вызывают `applyChange(change)`. `ListScreen`
прокидывает им `applyChange` вместо `onSynced`.

### Флаш = уже diff-синк

Флаш — один вызов `syncItems(listId, { lastSeenSeq, changes })`, где
`changes` — все правки из очереди (эндпоинт уже принимает батч). Протокол
`sync` — delta-pull (`items` с `version > lastSeenSeq`), а `lastSeenSeq`
закеширован вместе с `items`. Поэтому «сравнить кеш с сервером и обновить
при различиях» — это ровно то, что уже делает `mergeItems(cached,
response.items)`; отдельную diff-логику писать не нужно, флаш её и есть.

Триггеры флаша: маунт `ListScreen` (смена списка) и `window`-событие
`online`.

## Ретраи и ошибки

- Успешный флаш → `mergeItems` результата в `items`, `lastSeenSeq =
response.seq`, очередь очищается, кеш перезаписывается.
- Ошибка не `ApiError` (сеть недоступна) → очередь остаётся как есть, ждём
  следующий `online`.
- `ApiError` (сервер ответил и отклонил — например 404, список удалили пока
  были офлайн) → `attempts++` на все правки из этой попытки; после 3 неудач
  правка помечается `failed: true`, авто-ретраи для неё прекращаются.
  Ручного retry-действия в этой версии нет — если понадобится, отдельная
  задача.

## Холодный старт офлайн

`app/App.tsx`: bootstrap-эффект сначала пробует `GET /lists`+`GET
/categories` как сейчас. Если `fetch` падает не-`ApiError`-ошибкой (нет
сети) — читает `kupi:bootstrap` из `localStorage`. Если кеша тоже нет (самый
первый запуск устройства офлайн) — поведение не меняется (пустой экран) —
не покрытый край, а не регрессия: до этой задачи первый запуск офлайн не
работал вообще никак.

## Sync-статус

В `features/list-menu`, пункт, зарезервированный под это в
`2026-07-01-list-header-menu-design.md`. Производится из
`pendingCount`/`failedCount`/`navigator.onLine`:

| Условие                                   | Текст                 |
| ----------------------------------------- | --------------------- |
| `pendingCount === 0 && failedCount === 0` | «Синхронизировано»    |
| `pendingCount > 0 && online`              | «Синхронизация…»      |
| `pendingCount > 0 && !online`             | «Офлайн, N в очереди» |
| `failedCount > 0`                         | «N не отправлено»     |

## Тестирование

У `packages/client` пока нет тестраннера вообще (зафиксировано в
`CLAUDE.md`). Эта задача впервые вносит чистую, не завязанную на DOM логику,
которую стоит покрыть юнит-тестом: `applyChangeLocally` и ретрай-политику
очереди (по возможности вынести в чистый редьюсер, тестируемый без
React/DOM). Заводим `vitest` (единственный вменяемый вариант под
Vite/ESM-стек клиента — `node --test`, которым пользуется сервер, не умеет
транспилировать TS/JSX без отдельной обвязки) именно под эту задачу, не
заранее.

## Осознанно отложено

- Ручной retry для `failed`-правок.
- Фоновый флаш для списков, которые сейчас не открыты (флашим только
  активный список — при открытии другого списка его очередь флашнется
  тогда).
- Сглаживание визуального «прыжка» при stale-while-revalidate (скелетон/фейд
  при подмене кеша на актуальные данные).
- Background Sync API / IndexedDB — не поддержаны в Safari/iOS и добавляют
  сложность (async-хранилище, SW↔React синхронизация через `postMessage`)
  без выигрыша на этом объёме данных.
