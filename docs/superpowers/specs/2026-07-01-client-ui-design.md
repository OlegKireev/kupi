# Дизайн: клиентское приложение (экраны + визуальный стиль)

Дата: 2026-07-01
Статус: согласован, готов к плану реализации

## Краткое описание

Клиент (`packages/client`) сейчас — пустой каркас (React + Vite + `vite-plugin-pwa`,
только заглушка в `App.tsx`). Этот документ фиксирует IA/навигацию и визуальный стиль
для первой реальной реализации UI. Технические детали sync-движка на клиенте
(IndexedDB-схема, очередь pending changes, retry) в этот документ не входят —
отдельный спек, когда дойдём до реализации синхронизации.

Продуктовый контекст и решения по бэкенду — см.
`docs/superpowers/specs/2026-06-30-shopping-lists-design.md`. Ничего из решений
бэкенда этот документ не меняет.

## Ключевые решения

| Область           | Решение                                                               |
| ----------------- | --------------------------------------------------------------------- |
| Платформа         | Mobile-first; десктоп — тот же адаптивный layout, просто шире         |
| Навигация         | Один экран (активный список) + модалки/sheet поверх, без роутера      |
| Онбординг         | Без экрана-приветствия — сразу в пустой список с текстовой подсказкой |
| Профиль/участники | Без имён в MVP — только число участников                              |
| Подсказки товаров | Персональные (по account_id), как уже реализовано на бэке             |
| Стилизация        | Обычные `.css`-файлы, без Tailwind/CSS-in-JS                          |

## Экраны и навигация

Приложение — по сути один экран: активный список покупок. Всё остальное — модалки
и sheet поверх него, без отдельных роутов.

- **Шапка**: название текущего списка с `▾`. Тап открывает список списков
  (переключение + «Новый список» внизу). Список из одного элемента — это нормально
  и ожидаемо (по продукту пользователь обычно ведёт один список).
- **Иконка `⋮`** в шапке открывает меню:
  - Пригласить в список (генерит инвайт-код/ссылку)
  - Участники (N) — только число, без имён (см. «Профиль» ниже)
  - Подключить устройство (генерит link-код/ссылку)
  - Переименовать список
  - Удалить/покинуть список
  - Тихая строка статуса синка внизу того же меню (не баннер, не отдельный экран):
    «Синхронизировано только что» / «Не синхронизировано, N изменений в очереди»
    при офлайне.
- **Приглашение / привязка устройства (принимающая сторона)**: переход по
  diep-ссылке открывает экран с уже подставленным кодом; тот же экран позволяет
  ввести код вручную (единый UI для обоих способов, без отдельного пути для ручного
  ввода).
- **Роутинг**: без `react-router`. При загрузке — ручной разбор `window.location`
  на invite-/link-код из ссылки; переключение модалок/sheet — обычный `useState`.
  Экранов, которые нужно было бы адресовать URL, помимо этого diep-линка, нет.

### Онбординг

Первый запуск (аккаунт создаётся молча, без регистрации) сразу открывает пустой
список «Мои покупки» — без отдельного экрана-приветствия. Пустое состояние
показывает короткую текстовую подсказку под полем ввода, например:

> Список пуст. Начни печатать выше — появятся подсказки из твоих частых покупок.

### Профиль / участники

В модели данных нет профиля: `accounts` — это только `id` + `createdAt`, без имени и
аватара (осознанное анонимное MVP). Экран «Участники» в MVP показывает только
**число** участников списка, без списка имён. Изменений в бэкенде не требует.

## Визуальный стиль

Гибрид минимализма и тёплого стиля: светлый фон вместо стерильно-белого, тонкие
границы вместо теней, скруглённые строки-карточки вместо тяжёлых блоков.

- **Фон**: тёплый белый `#fdfcfa`.
- **Акцент**: приглушённый терракотовый `#c9764f` (чекбоксы, активные элементы,
  деструктивные текстовые действия вроде «Удалить»).
- **Строки списка**: без теней, скруглённые углы (~10px), чередующаяся лёгкая
  подложка (`#f7f4ef`) для читаемости вместо разделительных линий.
- **Категория товара**: эмодзи-иконка берётся из уже существующего
  `category.icon` (см. `packages/server/src/db/schema.ts` — пресетные категории уже
  сидятся с иконкой и цветом) + маленькая цветная точка `category.color` справа в
  строке. **Без текстовой подписи категории** в строке — полное название видно
  только в пикере категории (раскрытая строка). Товар без категории
  (`categoryId: null`) — вообще без иконки/точки, никакого fallback-значка.
- **Типографика**: системный шрифтовый стек (без подключения веб-шрифтов), обычный
  текстовый вес для товаров, полужирный для заголовка списка.

## Взаимодействие с товаром

- **Чекбокс** — тап отмечает/снимает отметку, отмеченный товар — зачёркнутый текст,
  приглушённый цвет.
- **Тап по строке** (не по чекбоксу) разворачивает её на месте — инлайн-редактор:
  - Степпер количества (`−` / число / `+`).
  - Чипы категорий (горизонтальный список, тап выбирает/меняет категорию товара).
  - Текстовая ссылка «Удалить товар» внизу раскрытой строки.
- **Свайп по строке** — альтернативный (не единственный) способ удаления, в
  дополнение к ссылке в раскрытой строке, а не вместо неё.
- **Добавление товара**: поле ввода вверху экрана. По мере набора — выпадающий
  список подсказок из `GET /suggestions?q=` (сортировка по частоте использования,
  уже реализована на бэке, персонально по аккаунту — см. таблицу решений). Каждая
  подсказка показывает свою иконку категории и счётчик использований. Выбор
  подсказки создаёт товар **с категорией найденного совпадения** (не нужно
  категоризировать вручную). Вариант «Добавить как новый товар» — без категории,
  как обычно. Добавленный товар появляется **вверху** списка (видимое
  подтверждение действия), а не в конце.

## Осознанно отложено

- Кастомные имена/аватары участников списка.
- Общие (не персональные) подсказки на уровне списка вместо аккаунта.
- Полноценный desktop-layout с несколькими колонками.
- Технические детали клиентского sync-движка (IndexedDB-схема, очередь, retry,
  обработка конфликтов на UI) — отдельный спек при реализации синхронизации.
